# PRIORITISED IMPROVEMENT RECOMMENDATIONS

## 1. MISSING FUNCTIONS — Critical Capability Gaps

**A. `get_function_body`** (HIGH IMPACT)
- **Rationale:** The agent can list functions and upsert them, but cannot inspect existing function bodies. This breaks debugging, auditing, and iterative refinement workflows. Mid-chain introspection requires this.
- **Spec:** Retrieve and return the full body of a named function from the functions table. Parameter: `setting_name` (function name).

**B. `delete_function`** (HIGH IMPACT)
- **Rationale:** ROSTER_NO_SILENT_DELETE harness requires logging before deletion, but there is no function to actually delete. The agent cannot honour this constraint. Must be paired with audit logging.
- **Spec:** Delete a function by name, log reason via run_bash_command to a deletion audit trail, update modified timestamp.

**C. `get_bash_log_filtered`** (MEDIUM IMPACT)
- **Rationale:** Current `get_bash_log` lacks filtering. No way to search by exit code, command pattern, or date range. For cron/daemon troubleshooting, this is essential.
- **Spec:** Extend with optional filters: `exit_code` (int), `command_pattern` (str, substring match), `since_minutes_ago` (int). Default to last 10 rows if no filters.

**D. `restart_service`** (MEDIUM IMPACT)
- **Rationale:** Agent manages models and reloads prompts but has no way to restart the Kobold or Ollama service itself. This is a gap for daemon deployment and self-recovery.
- **Spec:** Systemctl/supervisorctl restart for named service (e.g., `kobold`, `ollama`). Validate service name against whitelist. Log exit code and stderr.

**E. `health_check`** (MEDIUM IMPACT)
- **Rationale:** No built-in health probe for endpoints. For autonomous cron/daemon, the agent should poll ENDPOINT_KOBOLD and ENDPOINT_OLLAMA to detect stalls.
- **Spec:** HTTP GET to configured endpoints with 5s timeout. Return status code and latency. Log failures but do not block.

## 2. MISSING SETTINGS — Observability & Resilience

**A. `BASH_TIMEOUT_SECONDS`** (HIGH IMPACT)
- **Current:** Hard-coded 30s in run_bash_command.
- **Rationale:** Daemon and cron tasks have variable SLAs. Should be operator-configurable per deployment profile.
- **Spec:** Add to settings_values, default 30, range 5–300. Pass to run_bash_command at runtime.

**B. `HEALTH_CHECK_INTERVAL_SECONDS`** (MEDIUM IMPACT)
- **Rationale:** If health_check is implemented, agent needs to know how often to poll. Cron cannot schedule without this parameter.
- **Spec:** Add to settings_values, default 300 (5 min), range 30–3600.

**C. `LOG_RETENTION_DAYS`** (MEDIUM IMPACT)
- **Rationale:** No policy for agent_bash_logs table size. Long-running daemons will bloat the DB. Operator needs to set retention.
- **Spec:** Add to settings_values, default 7, range 1–90. Call cleanup on every Nth bash execution or via scheduled task.

**D. `ERROR_ESCALATION_WEBHOOK`** (LOW IMPACT, HIGH VALUE)
- **Rationale:** For daemon mode, silent failures are dangerous. Agent should be able to POST structured alerts to an external endpoint (Slack, PagerDuty, webhook).
- **Spec:** Add to settings_values (optional, empty string = disabled). Trigger on consecutive bash failures or health check failure.

**E. `DRY_RUN_MODE`** (MEDIUM IMPACT)
- **Rationale:** Agent can modify DB and run commands. Operator needs a safe testing mode that logs intended actions without executing them.
- **Spec:** Add to settings_boolean. When set to 1, run_bash_command and destructive DB calls log their intent but do not execute.

## 3. HARNESS GAPS — Under-Specified Constraints

**A. Strengthen `BASH_NO_DESTRUCTIVE_FLAGS`** (HIGH IMPACT)
- **Current:** Lists specific tools but no regex/pattern matching. An operator could bypass with `rm -rfv` or `/bin/rm -rf`.
- **Rationale:** Harness should block command patterns, not just string literals.
- **Spec:** Expand to: block any command containing regex `(^|\s)(rm|dd|mkfs|shred|wipefs|fdisk|parted)(\s|$)` with `-[rRfF]*`; improve error message to include matched pattern.

**B. Add `BASH_NO_NETWORK_EXPOSURE`** (MEDIUM IMPACT)
- **Rationale:** No restriction on network commands. Daemon could exfiltrate data via curl/wget/nc if compromised.
- **Spec:** Never allow `curl`, `wget`, `nc`, `socat`, `scp` in run_bash_command unless whitelisted by URL pattern (e.g., internal registry). Log intent and deny by default.

**C. Add `FUNCTION_SCHEMA_VALIDATION`** (MEDIUM IMPACT)
- **Rationale:** upsert_function accepts arbitrary Python. No validation that body is safe or well-formed.
- **Spec:** Before upsert, parse as AST and reject if it contains `import`, `exec`, `eval`, `subprocess`, `__import__`, or open file handles. Log rejection reason.

**D. Clarify `SCHED_MAX_STAGGER_DELAY`** (LOW IMPACT)
- **Current:** References a `/stagger` directive that does not appear in the function roster.
- **Rationale:** Incomplete. Either remove or implement the stagger scheduling function.
- **Spec:** If intended, add `stagger_task` function; if not, remove this harness rule and document in a deprecation note.
